# VFX capture and flash-audit evidence

This directory is the visual evidence loop for the attack-VFX production plan. A visual change is not ready for review until it has real gameplay stills, a video/contact-sheet strip, and a luminance report generated by the commands below.

## Run a phase capture

From `apps/web-toy`:

```bash
npm run vfx:capture -- --iteration p1-motion
```

The normal command is intentional: it starts a fixed-seed Greg run with `?autopilot=1&hero=greg&seed=3`, captures at 5, 30, 90, and 180 **simulation seconds**, and makes the first visible upgrade choice through the actual DOM button. This keeps a normal-speed run progressing without relying on a simulation-only shortcut.

Its output is `docs/vfx/captures/<iteration>/`:

- `still-5s.png`, `still-30s.png`, `still-90s.png`, and `still-180s.png`
- two 10-second `videos/clip-*.webm` files beginning near 30s and 180s
- color and grayscale contact sheets for each clip
- a per-clip `flash-audit-*.json` and top-level `report.json`

Extracted PNG frames are used for the audit and then discarded by default to keep the repository small. Use `--keep-frames` when an investigator needs the raw strips; the retained video is the canonical replay source.

## Fast plumbing smoke test

```bash
npm run vfx:capture -- --quick --headless --iteration local-smoke
```

`--quick` produces 2s/5s evidence and one 2-second clip. It proves the harness works; it is not a substitute for a phase report. `--stress` is also a plumbing option only: its accelerated simulation changes the apparent number of render frames in short-lived effects, so it must not be used to score visual quality.

## Flash audit

The capture invokes `flash-audit.mjs` automatically. It converts every captured frame to an 8x8 linear-relative-luminance grid and fails a cell if it has more than three reversals larger than 0.10 in any rolling second.

To audit retained extracted frames directly:

```bash
npm run vfx:flash-audit -- --frames docs/vfx/captures/<iteration>/frames/clip-180s --fps 20 --out report.json
```

Use `--flash-only` when the P6 gate needs a new 180-second clip and a nonzero exit code on a flash failure:

```bash
npm run vfx:capture -- --flash-only --iteration flash-gate
```

## Browser and boot fallbacks

The capture tries headed Chromium first, because a real GPU path is preferred. If the environment cannot open a headed window, it automatically retries headless Chromium with SwiftShader; `report.json` records the fallback. If the dev build cannot reach a running WebGL combat frame, the script builds and retries with Vite preview. A failure after both paths creates `BLOCKED.md` in the requested capture directory rather than silently claiming a visual pass.

You can force a specific recovery route:

```bash
npm run vfx:capture -- --preview --headed --iteration retry-preview
npm run vfx:capture -- --headless --iteration retry-swiftshader
```

The canonical P0 baseline belongs in `baseline-2026-07-13/`; later phases create one named capture folder and a short scored report, not a collection of accidental reruns.
